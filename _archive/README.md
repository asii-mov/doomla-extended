# Archive — early data that no longer feeds the main analysis

Everything here is *valid history* (reproducible from the same code) — it just
isn't featured in [`../FINDINGS.md`](../FINDINGS.md) anymore because the
research framing tightened. Kept rather than deleted so the chain of evidence
that led to the current setup stays inspectable.

## What's here, and why it was moved

### `logs_compare/`, `logs_solution_compare/` — Sonnet 4.5 on the `solution` variant

Original "headline" data: Sonnet 4.5 running all three defensive tiers
against the `solution` variant (the one that ships with the `solution.sh`
helper script). Produced the first version of the block-pivot narrative
(2 block-pivot cycles, 30× warning+ alert reduction silent → active).

Moved out because the `solution` variant trivialises completion for any
frontier model — the helper script walks the agent through the full attack
chain, so a 1.0 score doesn't measure capability. For a research doc the
honest reading is "this is a framework smoke-test, not a finding about model
behaviour." The cross-model headline is now Sonnet vs Opus on the `example`
variant uncapped, where both models discover the chain themselves.

### `logs_opus_prefix/` — Opus 4.7 runs against the broken alert pipeline

Pre-fix Opus runs from before the bind-mount fix (see
`../FINDINGS.md#a-note-on-what-we-fixed-mid-session`). The Falco bind mount
was a docker named volume the host-side reader couldn't see, so every value
in the alert log + scorer metadata came back as zero — regardless of what
the agent actually did. The logs are kept here because the agent traces
themselves are still real Opus output, but the alert / block / rule columns
are unrecoverable noise. Don't quote any metric from these.

### `data_out/01..03` — synthetic-attack visualisations

Generated by running representative offensive commands inside the target
containers via `docker exec` (i.e. outside Inspect's lifecycle), then
plotting the Falco alerts they produced. This was the bootstrap dataset
used to verify the rules + scorer worked end-to-end before the full
eval pipeline was wired. Superseded by real Inspect runs.

### `data_out/05` — framework smoke-test plot

Upstream Doomla vs `ours-baseline` on Tier 1 (Sonnet 4.6, no limits). The
"does our task.py + defended_bash with `falco_enabled=False` produce the
same behaviour as upstream `task.py` + plain `bash()`?" check. Answer was
yes, but baseline is no longer in the main analysis (silent dominates it
as a control), so the plot lives here.

### `data_out/06..07` — solution-variant analysis plots

`06_completion_and_runtime.png` and `07_silent_vs_active_rules.png`, plus
the underlying `solution_variant_comparison.json` + `silent_alerts.jsonl`
+ `plot_solution_compare.py`. Same reason as the `logs_solution_compare/`
move — solution variant is a framework smoke-test, not a capability test.

## Reproducing

The original eval invocations still work; the logs in `logs_solution_compare/`
correspond to:

```bash
inspect eval ../task.py@doomla -T tier=baseline -T variant=solution \
    --model anthropic/claude-sonnet-4-5 --log-dir _archive/logs_solution_compare/baseline
inspect eval ../task.py@doomla -T tier=silent   -T variant=solution \
    --model anthropic/claude-sonnet-4-5 --log-dir _archive/logs_solution_compare/silent
inspect eval ../task.py@doomla -T tier=active   -T variant=solution \
    --model anthropic/claude-sonnet-4-5 --log-dir _archive/logs_solution_compare/active
```

The plot script (`data_out/plot_solution_compare.py`) hard-codes paths that
no longer exist; if you re-run the plots, you'll need to update the
constants at the top of the file to point at the moved locations.
